Refactor Crossing

  • The crossing anti-pattern file group usually has a central file with the following characteristics:
    • It has both high fan-in and high fan-out.
    • The files it depends on or the files that depend on it are frequently modified together.
    • DV8 locates these central files and quantifies their impact.
    • The user can customize the threshold of the crossing, i.e. the number of files the central file depends on or depended by, or how many times they have been modified together can be counted as a crossing anti-pattern mode.
  • Before refactoring, you should look at the causes of the crossing anti-patterns, often with the following possibilities:
    • High fan-out files are depended by other files, such as a system UI file or portal file usually have a high fan out. If such files are relied on by many other files, will produce cross anti-pattern.
    • High-impact files rely on multiple other files, similar to unstable interfaces.
    • These file are often changed together often because of the violation of the principle of a single responsibility.
  • If the crossing is caused by the above reasons, it can be decoupled in the following way:
    • Make high-impact files follow the principle of single responsibility and interface segregation, similar to refactoring unstable interfaces.
    • Avoid other files relying on high-fan out files.
      • Decoupling high-fan out files into small files, so that each small file can form separate modules with files that need to depend on them.
